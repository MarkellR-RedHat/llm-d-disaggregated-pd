Deploying llm-d with Disaggregated Prefill and Decode: Separate the Phases, Double the Throughput

By Markell Rawls, Developer Advocate, Red Hat

Large language model inference is a tale of two phases. First comes prefill, where the model processes your entire input prompt in parallel, crunching through thousands of tokens simultaneously. Then comes decode, where the model generates its response one token at a time, methodically reading from memory to produce each word. These phases have fundamentally different computational profiles. Prefill is compute-bound, saturating the GPU's arithmetic units with massive matrix multiplications. Decode is memory-bandwidth-bound, spending most of its time waiting for data to arrive from GPU memory rather than actually computing. When both phases run on the same GPU, they interfere with each other, and neither runs optimally.

This is where llm-d comes in. As a CNCF Sandbox project, llm-d enables disaggregated prefill and decode, running each phase on dedicated hardware optimized for that specific workload. Built on top of vLLM and designed to run on OpenShift and Kubernetes, llm-d gives you production-ready infrastructure for deploying LLMs at scale. In this post, we'll walk through deploying Llama 3.3 70B with disaggregated prefill and decode, showing you the performance characteristics, deployment configuration, and real-world benchmark data.

Understanding the Two Phases

Let's break down what actually happens during prefill and decode, because understanding the mechanics is key to understanding why disaggregation works.

Prefill takes the entire input prompt and processes all tokens simultaneously through the transformer layers. This is a large matrix multiplication that fully utilizes GPU compute cores. For a 1,000-token prompt on Llama 3.3 70B, this means processing all 1,000 tokens through 80 transformer layers in parallel. The GPU's tensor cores are running at full speed, performing trillions of floating-point operations per second. This phase is FLOPS-bound—the limiting factor is how fast the GPU can multiply matrices, not how fast it can read from memory.

Decode generates output tokens one at a time. Each new token requires reading the entire KV cache (the attention keys and values from all previous tokens) from GPU memory. This is memory-bandwidth-bound. The GPU cores are mostly idle, waiting for data to arrive from HBM (High Bandwidth Memory). Even on H200 GPUs with 4.8 TB/s of memory bandwidth, this is still the bottleneck. The compute units could process the data much faster than the memory subsystem can deliver it.

The interference problem: When a GPU is doing prefill for one request and decode for another, the prefill computation saturates the compute units while decode needs memory bandwidth. But they share the same physical GPU resources. The prefill work queues up compute operations that block the decode's memory reads. The decode work consumes memory bandwidth that could be used to feed data to the prefill operations. Neither phase runs optimally, and you get worse performance than if you ran them sequentially.

Baseline Performance: Why Caching Matters

Before we talk about disaggregation, we need to understand KV cache reuse, because it's the foundation that makes everything else possible. When you process a prompt, the model computes attention keys and values for every token at every layer. These KV values are stored in the cache. If another request comes in with the same prompt prefix, you can reuse those cached KV values instead of recomputing them. This saves all the prefill work for those tokens.

Here's what KV cache reuse looks like in practice, measured as time-to-first-token at different context lengths:

Context Length Cold TTFT Cached TTFT Improvement
500 tokens 0.65s 0.16s 4.0x
1,000 tokens 0.24s 0.16s 1.5x
4,000 tokens 0.45s 0.16s 2.8x
8,000 tokens 0.93s 0.25s 3.8x
16,000 tokens 1.90s 0.19s 10.0x
32,000 tokens 4.74s 0.24s 19.8x

At 32,000 tokens, cached inference delivers TTFT in 0.24 seconds—a 19.8x improvement over cold inference at 4.74 seconds. The longer the context, the more caching matters.

But caching only helps if requests actually hit the same pod that has the cache. With naive round-robin routing, requests land on random pods, and each pod builds its KV cache from scratch. Cache reuse drops to 0.6-1.0x—effectively no benefit. This is where intelligent routing comes in, and why disaggregation needs to be paired with cache-aware routing.

As you scale concurrency, throughput increases linearly until you saturate the GPU's memory bandwidth:

10 359.8
50 1,483.4

Deploying Disaggregated Prefill and Decode

Now let's look at how to actually deploy this. The llm-d API makes it straightforward. You start with a standard LLMInferenceService (replicas running both prefill and decode), then add dedicated prefill workers, enable KV cache transfer, and configure disaggregated routing.

Here's the complete configuration for Llama 3.3 70B with disaggregated prefill and decode:

apiVersion: inference.llm-d.ai/v1alpha1
kind: LLMInferenceService
metadata:
  name: llama-3-3-70b-fp8
  namespace: llm-d
spec:
  modelArtifact:
    uri: "oci://quay.io/llm-d/models:llama-3.3-70b-instruct-fp8-kv-int8"
  inferenceConfig:
    replicas: 2
    accelerator:
      type: nvidia.com/gpu
      count: 4
      product: H200
  prefill:
    replicas: 1
    accelerator:
      type: nvidia.com/gpu
      count: 4
      product: H200
  kvCacheTransfer:
    connector: nixl
    transport: rdma
  routing:
    strategy: disaggregated-prefill-decode
    prefillRouting:
      loadBalancing: least-load
    decodeRouting:
      loadBalancing: kv-affinity

Let's break down each critical section:

prefill.replicas: This creates dedicated prefill pods that only handle prompt processing. These pods don't do any token generation—they just run the initial forward pass through the model and transfer the KV cache to a decode pod.

kvCacheTransfer: This enables NixL (NVIDIA's efficient KV cache transfer protocol) for zero-copy KV cache transfer over RDMA. When a prefill pod finishes processing a prompt, it transfers the KV cache directly to a decode pod's GPU memory without going through CPU memory or the network stack. RDMA (Remote Direct Memory Access) means the GPU can write directly to another GPU's memory over the network, bypassing the host CPU entirely.

routing.strategy: disaggregated-prefill-decode: This tells llm-d to route new prompts to prefill pods and ongoing generation to decode pods. When a request comes in, the router sends it to a prefill pod. The prefill pod processes the prompt, transfers the KV cache, and the decode pod takes over for generation.

prefillRouting.loadBalancing: least-load: Distributes prefill work across available prefill pods based on their current load. This keeps prefill latency consistent even when traffic spikes.

decodeRouting.loadBalancing: kv-affinity: Routes decode to the pod that already has the KV cache. If you're doing multi-turn conversations or requests with common prefixes, this ensures cache reuse.

Expected Performance Impact

What does this actually buy you? Based on modeling and early production deployments, here's what to expect:

Throughput: 30-60% higher sustained throughput because prefill and decode pods can pipeline. While decode pods are generating tokens for batch N, prefill pods are processing prompts for batch N+1. This overlap means you're utilizing more of your total GPU capacity at any given moment. In a traditional deployment, a GPU can't start the next prefill until the current decode finishes. With disaggregation, those operations happen concurrently on different hardware.

TTFT stability: Dedicated prefill pods maintain consistent TTFT even under heavy decode load. In a traditional deployment, if you have 50 concurrent requests in their decode phase, a new prompt has to wait for compute resources. With dedicated prefill pods, new prompts get immediate attention because decode work is happening elsewhere.

GPU utilization: Both pod types run at higher efficiency. Prefill pods keep compute units busy with matrix multiplications. Decode pods optimize for memory bandwidth with larger batch sizes. Each pod type is tuned for its specific workload pattern. A traditional pod has to compromise—it can't optimize batch size for both prefill and decode simultaneously.

Honest Limitations

Disaggregated prefill and decode is not a universal win. Here are the real trade-offs:

KV cache transfer adds latency. Even with NixL and RDMA, transferring the cache is not instantaneous. For short prompts (under 1,000 tokens), the transfer overhead can exceed the prefill time savings. This is most beneficial for long-context workloads where prefill time dominates.

Requires more total GPU memory. You have separate pods for each phase, which means the model weights are loaded twice (once on prefill pods, once on decode pods). If you're running at the edge of GPU memory capacity, this may not be feasible without model quantization or smaller batch sizes.

Most beneficial at scale. For 1-2 pods serving low traffic, a standard deployment is simpler and performs just as well. Disaggregation shines when you have enough traffic to keep dedicated prefill pods busy and enough decode work to benefit from the separation.

Network bandwidth between pods matters. For large KV caches (32k+ context), you're transferring gigabytes of data between pods. RDMA over InfiniBand or RoCE gives you 100+ GB/s, but standard Ethernet networks will bottleneck. Make sure your infrastructure can handle the bandwidth.

What's Next

llm-d is under active development as a CNCF Sandbox project. The roadmap includes dynamic scaling of the prefill/decode ratio based on workload patterns, integration with LoRA adapter routing for multi-model serving, and improved cache transfer protocols. The project is open source and community contributions are welcome.

If you're deploying LLMs at scale on Kubernetes, llm-d gives you production-ready infrastructure with advanced features like disaggregated prefill/decode, prefix-aware caching, and GPU-aware routing. It's built on proven components (vLLM, Kubernetes, OpenShift) and designed for operational simplicity.
